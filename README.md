# Stash

An encrypted blob storage engine for [ShrouDB](https://github.com/shroudb/shroudb).

## What It Does

Stash stores blobs in S3-compatible object stores with envelope encryption. Blobs are encrypted locally using a data encryption key (DEK) generated by Cipher. The DEK is wrapped by Cipher's keyring and stored alongside the blob metadata. The blob never touches S3 in plaintext.

```
STASH STORE my-doc <base64_data> CONTENT_TYPE application/pdf
→ { "status": "ok", "id": "my-doc", "keyring": "stash-blobs", "key_version": 1 }

STASH RETRIEVE my-doc
→ [metadata_json, plaintext_bytes]

STASH INSPECT my-doc
→ { "status": "ok", "id": "my-doc", "blob_status": "active", "plaintext_size": 12345, ... }

STASH REVOKE my-doc
→ { "status": "ok", "id": "my-doc", "revoke_mode": "hard" }
```

## Wire Protocol (RESP3)

### Store a blob

```
STORE <id> <data_b64> [KEYRING <name>] [CONTENT_TYPE <mime>] [CLIENT_ENCRYPTED <wrapped_dek>]
```

- `id` — unique blob identifier
- `data_b64` — base64-encoded blob data
- `KEYRING` — Cipher keyring for envelope encryption (default: `stash-blobs`)
- `CONTENT_TYPE` — MIME type metadata
- `CLIENT_ENCRYPTED` — passthrough mode: client already encrypted the data, provide the wrapped DEK

### Retrieve a blob

```
RETRIEVE <id>
```

Returns a RESP3 Array of two BulkStrings: `[metadata_json, plaintext_bytes]`.

For client-encrypted blobs, returns the raw ciphertext + wrapped DEK in metadata (client decrypts).

### Inspect metadata

```
INSPECT <id>
```

Returns blob metadata without downloading or decrypting. No S3 access, no Cipher interaction.

### Revoke a blob

```
REVOKE <id> [SOFT]
```

- **Hard (default):** Crypto-shred — destroy wrapped DEK, delete all S3 objects (master + viewer copies), tombstone metadata.
- **Soft (`SOFT` flag):** Mark as revoked. Sentry blocks future RETRIEVE. Blobs and DEKs preserved.

### Operational

```
HEALTH
PING
COMMAND
```

## Moat Integration

In Moat (the unified ShrouDB hub), Stash commands are prefixed:

```
STASH STORE my-doc <data_b64>
STASH RETRIEVE my-doc
STASH INSPECT my-doc
STASH REVOKE my-doc SOFT
```

### Configuration

```toml
[engines.stash]
enabled = true
bucket = "my-encrypted-blobs"
region = "us-east-1"
endpoint = "https://minio.internal:9000"  # optional, for S3-compatible services
keyring = "stash-blobs"                   # optional, default: stash-blobs
s3_key_prefix = "stash/"                  # optional prefix for S3 keys
```

## Architecture

```
shroudb-stash-core/     — BlobMetadata, BlobStatus, StashError, ViewerRecord
shroudb-stash-engine/   — StashEngine, ObjectStore trait, S3 + InMemory impls, local AES-256-GCM
shroudb-stash-protocol/ — RESP3 command parsing, dispatch, response types
```

## Cipher-less Mode

When Cipher is not available, Stash operates as a raw passthrough to S3 — blobs are uploaded unencrypted. Metadata tracking, access control (Sentry), and audit logging (Chronicle) still apply. This is the same graceful degradation pattern used by Sigil when its capabilities are absent.

## Security

- Blobs encrypted with AES-256-GCM when Cipher is available (random 12-byte nonce per blob)
- DEKs generated by Cipher, wrapped with the keyring's active key version
- Plaintext DEK held in `SensitiveBytes` — zeroized on drop
- Hard revoke is crypto-shredding: destroying the DEK makes the S3 ciphertext unrecoverable
- Without Cipher, data is stored unencrypted — use client-encrypted mode for encryption without Cipher
- Sentry ABAC policy enforcement on all operations
- All operations audited via Chronicle
