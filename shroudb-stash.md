## Role in Platform

Stash is the blob storage engine for ShrouDB. It encrypts arbitrarily-sized binary objects (files, images, documents) with AES-256-GCM using per-blob data encryption keys from Cipher, then uploads ciphertext to S3-compatible object stores. Metadata (wrapped DEKs, S3 locations, status) lives in the ShrouDB Store. Without Stash, ShrouDB has no binary large object capability — the other engines handle structured key-value data, not opaque blobs.

## Behavioral Surface

### Public API

**RESP3 wire commands** (6 crates, ~4,900 lines of source + tests):

| Command | ACL | Description |
|---------|-----|-------------|
| `STORE <id> <data_b64> [KEYRING] [CONTENT_TYPE] [CLIENT_ENCRYPTED <wrapped_dek>]` | write `stash.<id>` | Encrypt + upload blob |
| `RETRIEVE <id>` | read `stash.<id>` | Download + decrypt blob (returns RESP3 Array: `[metadata_json, blob_bytes]`) |
| `INSPECT <id>` | read `stash.<id>` | Metadata-only read (no S3 access, no Cipher call) |
| `REVOKE <id> [SOFT]` | write `stash.<id>` | Hard (crypto-shred) or soft revocation |
| `AUTH <token>` | none | Authenticate connection |
| `HEALTH` / `PING` / `COMMAND` | none | Operational |

**Embedded library API** (`StashEngine<S: Store>`):
- `store_blob(StoreBlobParams)` / `retrieve_blob(id, actor)` / `inspect_blob(id, actor)` / `revoke_blob(id, soft, actor)`
- Capabilities injected at construction: `StashCipherOps`, `PolicyEvaluator`, `ChronicleOps` — all optional.

**Client SDK** (`StashClient`): typed Rust async client with `connect()`, `auth()`, `store()`, `store_client_encrypted()`, `retrieve()`, `inspect()`, `revoke()`.

**CLI** (`shroudb-stash-cli`): interactive REPL or single-command mode. Supports `@filepath` for file upload. Connects to server via TCP.

### Core operations traced

**STORE flow** (`engine.rs:96-195`):
1. Duplicate check via `load_metadata(id)` — fails with `AlreadyExists`
2. ABAC policy check via Sentry (`check_policy`)
3. Three encryption paths:
   - **Client-encrypted**: passthrough — client provides `wrapped_dek`, Stash stores blob as-is
   - **Server-encrypted** (default): `cipher.generate_data_key(256)` → `encrypt_blob(dek, plaintext)` (AES-256-GCM) → upload ciphertext to S3
   - **No Cipher**: raw upload (warns in logs)
4. S3 `put()` → persist `BlobMetadata` + empty `ViewerMap` to Store → emit Chronicle audit event

**RETRIEVE flow** (`engine.rs:206-259`):
1. Load metadata → check status (Shredded/Revoked → error, Active → continue)
2. ABAC policy check
3. Download from S3
4. Decrypt path depends on metadata: client-encrypted → return raw + `wrapped_dek`; no DEK → return raw; has DEK → `cipher.unwrap_data_key()` → `decrypt_blob()` → return plaintext
5. `SensitiveBytes` auto-zeroizes plaintext DEK on drop

**REVOKE (hard)** (`engine.rs:295-364`):
1. Load metadata + viewer map
2. ABAC policy check
3. Delete all viewer S3 objects (cascade), then master S3 object
4. Clear `wrapped_dek` string (crypto-shred: DEK destroyed, AES-256-GCM ciphertext computationally unrecoverable)
5. Set status = `Shredded`, persist tombstone metadata, clear viewer map
6. Emit `REVOKE_HARD` audit event

### Capability gating

`Capabilities` struct (`capabilities.rs:48-53`):
- `cipher: Option<Box<dyn StashCipherOps>>` — envelope encryption. Without it: raw passthrough to S3.
- `sentry: Option<Arc<dyn PolicyEvaluator>>` — ABAC policy. Without it: open mode (all operations permitted).
- `chronicle: Option<Arc<dyn ChronicleOps>>` — audit trail. Without it: no audit events.

All three are optional — Stash degrades gracefully. No feature flags gate commercial behavior; the fence is capability injection at runtime.

## Cryptographic Constructs

| Construct | Detail |
|-----------|--------|
| **Blob encryption** | AES-256-GCM via `ring::aead` (`crypto.rs:17-89`) |
| **Nonce** | Random 96-bit (12 bytes) per blob via `ring::rand::SystemRandom` |
| **Ciphertext format** | `nonce (12B) \|\| ciphertext \|\| tag (16B)` — 28 bytes overhead |
| **DEK** | 256-bit, generated by Cipher's `GENERATE_DATA_KEY` |
| **Key wrapping** | Delegated to Cipher — Stash receives `CiphertextEnvelope` (base64-encoded wrapped DEK) |
| **Plaintext DEK handling** | `SensitiveBytes` from `shroudb-crypto` — zeroized on drop, never persisted |
| **AAD** | Empty (no additional authenticated data) |
| **Crypto-shredding** | Hard revoke clears `wrapped_dek` field — AES-256-GCM ciphertext without key is computationally unrecoverable. S3 object deletion is defense-in-depth. |
| **Key rotation** | Not implemented in Stash — `key_version` tracked in metadata but no re-wrap operation exists |

No HKDF, PBKDF2, or key derivation in Stash itself. All key management delegated to Cipher.

### Sentry / ACL integration

Two-layer authorization:

1. **Wire-level ACL** (`shroudb_acl::check_dispatch_acl`): checked in dispatch before engine call. Namespace-scoped: `stash.<id>` with Read/Write scope. Uses `AclRequirement` from `shroudb-acl`.
2. **Engine-level ABAC** (`check_policy`, `engine.rs:429-470`): if Sentry capability present, builds `PolicyRequest` with principal/resource/action and evaluates. Deny → `StashError::AbacDenied`.

Without Sentry: wire ACL still enforced via `shroudb-acl`; engine-level policy is open.

## Store Trait

Stash does **not** implement the Store trait — it **consumes** it. `StashEngine<S: Store>` is generic over any Store implementation.

- **Metadata storage**: `stash.meta` namespace (JSON-serialized `BlobMetadata`)
- **Viewer storage**: `stash.viewers` namespace (JSON-serialized `ViewerMap`)
- **Supported backends**: anything implementing `shroudb_store::Store`. In standalone mode: `EmbeddedStore` from `shroudb-storage`. In Moat: whatever Store backend Moat provides.
- **Blob storage**: separate `ObjectStore` trait — `S3ObjectStore` (production) or `InMemoryObjectStore` (testing). Not related to the Store trait.

## Standalone Extractability

**Extractable as independent product:** Yes, with caveats.

Stash is a complete, self-contained encrypted blob storage service:
- Standalone binary (`shroudb-stash-server`) on port 6699
- Client SDK and CLI included
- S3-compatible backend (AWS, MinIO, R2, etc.)
- Can run without Cipher/Sentry/Chronicle (degrades to raw passthrough)

**What it loses without siblings:**
- Without Cipher: no encryption — just a metadata-tracking S3 proxy
- Without Sentry: no ABAC policy enforcement beyond wire-level ACL tokens
- Without Chronicle: no audit trail
- Without Moat: must deploy as separate process (no embedded multi-engine topology)

**What would need to change for standalone sale:**
- Bundle a simplified key management layer (or document integration with external KMS like AWS KMS, HashiCorp Vault)
- Replace the `StashCipherOps` trait with adapters for third-party KMS
- Wire-level auth exists but relies on `shroudb-acl` for token validation — would need standalone auth story

### Target persona if standalone

DevOps/platform teams needing client-side encryption for S3 with crypto-shredding capability. Compliance-driven organizations where AWS SSE-S3/SSE-KMS is insufficient (cloud provider holds keys). Healthcare (HIPAA), finance, legal hold workflows.

### Pricing model fit if standalone

Usage-based: per-blob stored or per-GB encrypted, with a base tier. The crypto-shredding and forensic watermarking (v0.2) features are the premium differentiators. Open core base + commercial encryption/compliance tier would work.

## Architectural Moat (Component-Level)

The moat is **platform-level, not component-level**. Stash alone is a well-implemented encrypted S3 proxy — non-trivial but reproducible by a competent team in weeks. What makes it defensible:

1. **Envelope encryption integration with Cipher**: the `StashCipherOps` abstraction and the DEK lifecycle (generate → wrap → store → unwrap → zeroize) is clean but its value comes from Cipher's keyring management, not from Stash.
2. **Crypto-shredding with provable destruction**: clearing the wrapped DEK makes AES-256-GCM ciphertext computationally unrecoverable. This is a genuinely useful compliance primitive. The implementation is ~20 lines but the design decision and correctness guarantee are valuable.
3. **Viewer fingerprinting scaffolding** (v0.2): `ViewerMap`/`ViewerRecord` structures for per-viewer watermarked copies with independent DEK wrapping — positioned for leak tracing. Not yet implemented but the metadata model is ready.
4. **ABAC + audit integration**: two-layer auth (wire ACL + Sentry ABAC) with Chronicle audit trail gives compliance-ready access control. Again, value is platform-level.

Hard-won knowledge encoded: graceful degradation across three encryption modes (server/client/none), correct DEK zeroization lifecycle, viewer cascade deletion on hard revoke.

## Gaps and Liabilities

1. **No key rotation**: `key_version` is tracked in metadata but no re-wrap or re-encrypt operation exists. Blobs encrypted with a rotated-away key version become stuck unless Cipher preserves old versions.
2. **Empty AAD in AES-256-GCM**: no additional authenticated data binding the ciphertext to the blob ID or metadata. A ciphertext could theoretically be transplanted to a different blob ID without detection.
3. **No streaming encryption**: blobs are fully buffered in memory for encryption/decryption. Multi-GB blobs (mentioned in ABOUT.md) would OOM the process.
4. **No tenant isolation**: blob IDs are flat. Multi-tenant deployments rely entirely on Sentry ABAC for isolation — no namespace partitioning at storage level.
5. **No deduplication**: identical blobs stored under different IDs produce separate S3 objects and DEKs.
6. **Viewer fingerprinting not implemented**: `ViewerMap` and `ViewerRecord` are defined and persisted but the FINGERPRINT and TRACE commands (v0.2) don't exist yet. The metadata scaffolding is unused.
7. **No CHANGELOG**: version history not tracked in a committed file.
8. **Test coverage gap**: no tests for concurrent hard revoke races, S3 failure mid-upload (partial write), or Cipher failure mid-operation.
9. **`client_encrypted` integrity**: Stash stores client-encrypted blobs without verifying the wrapped DEK is valid or the ciphertext is well-formed. A corrupted upload is not detectable until the client attempts decryption.

## Raw Signals for Evaluator

- **~4,900 lines total** across 6 crates — compact, focused codebase. Low maintenance burden.
- **v0.1.6** — early but functional. Core STORE/RETRIEVE/INSPECT/REVOKE complete and tested.
- **28 integration tests** including MinIO container tests, ACL tests, and raw passthrough mode. Solid for the scope.
- **Dependency on 6 internal crates** (`shroudb-acl`, `shroudb-crypto`, `shroudb-chronicle-core`, `shroudb-storage`, `shroudb-store`, `shroudb-protocol-wire`, `shroudb-server-tcp`, `shroudb-server-bootstrap`, `shroudb-client-common`) — tight coupling to the ShrouDB ecosystem despite being architecturally separable.
- **The README explicitly calls out the compliance gap** that Stash addresses: AWS SSE-S3/SSE-KMS keeps keys with the cloud provider. This is the sales pitch for regulated industries.
- **Crypto-shredding is the killer feature** — provable data destruction without relying on S3 lifecycle policies or deletion propagation. This is genuinely hard to get from off-the-shelf tools.
- **Docker + config story is production-ready** for standalone deployment. Docker entrypoint handles volume mount permissions.
- **No telemetry, no phone-home, no license enforcement** — pure open core. Commercial value extracted at the platform level through Moat bundling and Cipher dependency.
